SP #033: FP8 Types
=================

We propose to add 8-bit floating-point ("FP8") types in Slang. These smaller
floating-point types could be used to accelerate many machine learning kernels.

Status
------

Status: Design Review

Implementation:

Author: Nicolas Celik

Reviewer:

Background
----------

The "FP8 Formats For Deep Learning" whitepaper
(https://arxiv.org/pdf/2209.05433) defines two 8-bit floating-point formats to
accelerate deep learning training and inference: E4M3 (4-bit exponent and 3-bit
mantissa) and E5M2 (5-bit exponent and 2-bit mantissa).

Slang currently does not support any FP8 types in the language. The
floating-point types currently defined in the language are `half`, `float`, and
`double`, which represent the `binary16`, `binary32`, and `binary64` formats
defined in IEEE 754, respectively.

However, the `CoopVecComponentType` enum defined in `hlsl.meta.slang` already
contains cases named `FloatE4M3` and `FloatE5M2`. The following functions
support `CoopVecComponentType::FloatE4M3` and `CoopVecComponentType::FloatE5M2`
as values for their `inputInterpretation`, `matrixInterpretation`, and
`biasInterpretation` parameters when targeting SPIR-V, HLSL, or OptiX:

- `coopVecMatMul`
- `coopVecMatMulAdd`
- `coopVecMatMulAddPacked`
- `coopVecMatMulPacked`
- `coopVecOuterProductAccumulate`

This support is only possible because these functions simply wrap around SPIR-V
instructions, HLSL builtins, and OptiX functions, depending on the target. When
the target is other than SPIR-V, HLSL, or OptiX, pure Slang implementations of
these functions are used. Because E4M3 and E5M2 are not supported in the
language, these pure Slang implementations currently do not support
`CoopVecComponentType::FloatE4M3` or `CoopVecComponentType::FloatE5M2` being
passed for `inputInterpretation`, `matrixInterpretation`, or
`biasInterpretation`. Additionally, FP8 cooperative vectors and cooperative
matrices cannot be created at all, as FP8 `CoopVec` and `CoopMat` types cannot
be represented.

Related Work
------------

E4M3 and E5M2 have been implemented in the following relevant APIs, languages,
IRs, and ISAs:

- in **OptiX, Vulkan, and HLSL,** where they are only supported as
  component/element types for cooperative vectors and cooperative matrices,
- in the **CUDA Math API,** where they can only be used in conversion
  operations,
- in **PTX and SPIR-V,** where they can be used in memory (load/store)
  operations, conversion operations, and (cooperative) matrix multiply-add
  operations, as well as in cooperative vector operations in the case of SPIR-V.

Note that scalar arithmetic on FP8 types is not supported in any of these APIs
and languages.

The current state of GLSL is similar to Slang's: E4M3 and E5M2 are not supported
in the language, but the `GLSL_NV_cooperative_vector` extension defines
constants named `gl_ComponentTypeFloatE4M3NV` and `gl_ComponentTypeFloatE5M2NV`
which can be used for the interpretation parameters of the `coopVec*` functions.

On x86-64, Intel AVX10.2 introduces instructions to convert vectors between FP8
(E4M3 and E5M2) and FP16 (Slang `half`). Intel AMX introduces instructions to
compute the dot product of E4M3 and/or E5M2 vectors and accumulate it into an
FP32 (Slang `float`) vector.

On AArch64, Armv9.2 introduces optional Advanced SIMD (Neon), SVE2, and SME2
instructions for E4M3 and E5M2. These instructions only operate on vectors and
matrices.

C, C++, Metal/MSL, and WebGPU/WGSL currently do not support FP8 types.

Proposed Approach
-----------------

We propose to define two new floating-point types in the Slang language:
`FloatE4M3` and `FloatE5M2` (the "FP8 types"), representing E4M3 and E5M2
respectively.

As FP8 types are mainly of interest for matrix operations in machine learning,
and there is no hardware support for scalar or general FP8 arithmetic, we
propose restricted support for FP8 types compared to other floating-point types:

- Support for explicitly constructing FP8 values with `FloatE4M3(x)` and
  `FloatE5M2(x)`, where `x` is any floating-point type, including `FloatE4M3`
  and `FloatE5M2`.
- No support for arithmetic, comparison, or math operations on FP8 types or
  `vector` types with FP8 element types.
- No support for implicit conversion of FP8 types to other floating-point types.
- Support for FP8 types as element types for `CoopVec` and `CoopMat`.

Detailed Explanation
--------------------

Unlike existing floating-point types, FP8 types would not implement any of the
following interfaces:

- `__BuiltinArithmeticType`
- `__BuiltinFloatingPointType`
- `__BuiltinRealType`
- `__BuiltinSignedArithmeticType`
- `IArithmetic`
- `IFloat`

As `CoopVec` and `CoopMat` currently require the element type `T` to implement
`__BuiltinArithmeticType`, we propose to define a new empty interface named
`ICoopVecElement` and change the definitions of `CoopVec`, `CoopMat`, and the
related `coopVec*` and `coopMat*` functions to use this new interface instead.
FP8 types would only implement `ICoopVecElement` and no other interface.

The following documentation should be updated:

- Section "Scalar Types" of `docs/user-guide/02-conventional-features.md`
- Section "Floating-Point Types" of `docs/language-reference/04-types.md`
